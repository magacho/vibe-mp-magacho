# Skill `codebase-360` — Auditoria 360° da Base de Código

> Handle: `vibe:codebase-360` · Foco: **diagnóstico executivo**, não correção automática.

Um único comando guarda-chuva que audita a base de código em **todas as
dimensões de uma vez** e entrega uma **apresentação HTML com a marca Bemobi**
(slide deck). Em vez de rodar quatro análises soltas, a `360` orquestra todas,
funde os achados num único schema e renderiza um deck com placar, resumo
executivo e regras de ouro.

A ideia central: *um raio-x da base que um líder técnico abre e entende em
minutos* — cada achado traz o problema, o custo de contexto para a IA, uma
sugestão, opções de solução avaliadas e o risco/impacto. É **report-only**:
nunca aplica mudanças sozinho.

Construída sobre o [`scip-query`](https://github.com/sourcegraph/scip). Se o
índice estiver ausente ou desatualizado, ela roda `scip-query reindex` antes
(ou guia a instalação via `scip-polyglot-setup`).

## Quando usar

- Você quer um **relatório completo / "raio-x" / review 360** da base de código.
- Precisa de uma **auditoria executiva** para apresentar a um time ou liderança.
- Quer **combinar várias análises** (AI/token, manutenção, reuso, componentes)
  num único deck em vez de rodar cada uma à parte.
- Quer mapear **onde a organização do código encarece o desenvolvimento com IA** —
  quantos tokens a IA precisa carregar para mexer com segurança em cada área.
- Quer extrair **regras de ouro** (estilo CLAUDE.md) que impeçam o código novo de
  recriar os mesmos problemas.

Gatilhos típicos: *relatório completo, raio-x, 360, auditoria executiva,
codebase audit, deck, apresentação, combinar análises, custo de AI, eficiência
de token, dead code, duplicação, reuso de componentes.*

## Quando NÃO usar

- Você quer uma análise **focada numa única dimensão** — use a skill específica
  (`scip-context-budget` para custo de AI, `scip-debloat`/`scip-maintainability`
  para manutenção, `scip-query`/`scip-explore` para exploração pontual).
- Você quer que as mudanças sejam **aplicadas automaticamente** — a `360` é
  report-only de propósito. Para aplicar e verificar, faça o handoff explícito
  para `scip-verify`.
- A base **não tem índice scip nem indexadores instalados** e você não quer
  instalá-los agora — o preflight vai pedir o setup antes de continuar.

## Como invocar

- **Claude Code / Cowork:** referencie o handle `vibe:codebase-360`, ou apenas
  descreva o pedido ("faz um raio-x da base", "quero um review 360", "monta um
  deck executivo da codebase") — a skill é disparada pela descrição.
- **claude.ai / Desktop (chat):** com a skill `codebase-360` instalada, peça a
  auditoria no repositório aberto.

## O que ela faz (as quatro lentes)

A `360` roda quatro seções independentes (paralelizáveis por subagentes) e funde
tudo num deck. Cada seção vira uma divisória + slides de achado:

| Seção | Lente | O que investiga |
|-------|-------|-----------------|
| 🤖 **AI / Eficiência de Token** | Como a organização e o tamanho dos artefatos inflam o custo de contexto da IA | Maiores arquivos, pior `edit-cost`, alto `rdeps`/`affected` (mudança não-cirúrgica), `deep-chains`, type/domain files gigantes |
| 🔧 **Manutenção & Dead Code** | Smells de arquitetura, políticas escondidas, código morto/bloat | `health`, `dead`, `isolated`, `stale-abstractions`, `cycles`, `drift` |
| ♻️ **Reuso de Código** | Funções/arquivos duplicados, costuras de consolidação | `similar`, `similar-files`, `similar-signatures`, `convergence` |
| 🎨 **Reuso de Componentes Visuais** | Componentes React duplicados, primitivos `ui/` subutilizados | `similar-files` em `components/**`, `hotspots`/`rdeps` em `components/ui/*` |

A seção de **AI** carrega o enquadramento de "custo de AI": cada achado diz
quantos tokens a IA precisa carregar para mexer naquela área — deixando visível
onde a má organização torna o desenvolvimento com IA caro e não-cirúrgico.

### O fluxo, em alto nível

| Fase | O que acontece |
|------|----------------|
| 0 — Preflight | Checa o ferramental (`scip-query`, `python3`, scip CLI + indexadores). Ajuda a instalar o que faltar antes de seguir. |
| 1 — Prepare | Garante o índice (`reindex` se preciso) e captura `stats` (documentos/símbolos) para o slide de título. |
| 2 — Run | Roda as quatro seções em paralelo, cada uma com seus comandos `scip-query` e a matemática de tokens compartilhada. |
| 3 — Merge | Funde as seções num JSON com `stats`, `scorecard` (uma nota por seção), resumo executivo e regras de ouro. |
| 4 — Render | Renderiza o deck HTML Bemobi: título → placar → resumo executivo → seções → "não recomendado". |
| 5 — Verify | (opcional) screenshot headless para conferência visual. |
| 6 — Golden Rules | (opt-in) oferece persistir as regras de ouro no `CLAUDE.md` do projeto analisado, só com confirmação explícita. |

O **resumo executivo** e as **regras de ouro** não são opcionais — todo run os
renderiza. Cada problema reportado ganha uma regra de ouro imperativa (voz de
CLAUDE.md) que impede o código futuro de recriar aquele problema.

## O que sai (o deck)

O artefato é um único HTML autocontido (`reports/codebase-360/AAAA-MM-DD.html`):

1. **Slide de título** — projeto, escopo, data, contagem de documentos/símbolos.
2. **Placar (scorecard)** — uma nota (A–F) por dimensão, com a manchete de cada uma.
3. **Resumo executivo** — uma linha por achado numa tabela de 6 colunas
   (Ação · Esforço · Impacto na operação · Impacto na change · Ganho · Regra de
   ouro), para enxergar a complexidade de relance.
4. **Seções** — divisória + slides de achado. Cada achado traz problema,
   evidência (`scip-query …`), custo de contexto, sugestão, opções de solução
   (Conservadora/Estrutural) com **prompt copiável** para implementar, e
   risco/impacto.
5. **"Não recomendado"** — mudanças tentadoras que custam mais conceitos do que
   removem.

As cores das seções seguem a paleta Bemobi e o logo é resolvido da skill
`bemobi-brand` (fallback em texto se ausente).

## Cenários onde a skill faz sentido

1. **Raio-x antes de um refactor grande.** Antes de investir num redesenho, a
   `360` mostra onde estão os custos reais (token, acoplamento, duplicação) e
   ordena por impacto ÷ risco — você ataca o que paga, não o que parece feio.

2. **Auditoria executiva para liderança.** O deck Bemobi com placar e resumo
   executivo é feito para ser apresentado: um líder técnico ou gestor entende o
   estado da base em minutos, sem ler código.

3. **"Por que a IA erra tanto nessa base?"** A seção de AI/Token quantifica o
   custo de contexto: quando um type-file gigante puxa 163 telas para qualquer
   edit, a IA não consegue ser cirúrgica. O deck mostra onde e quanto.

4. **Onboarding de um novo time/squad.** Um raio-x inicial dá o mapa de dívidas,
   código morto e duplicações — e as regras de ouro viram o guia do que não
   repetir.

5. **Estabelecer regras de ouro no CLAUDE.md.** Ao final, a skill oferece
   persistir as regras derivadas dos problemas no `CLAUDE.md` do projeto, para
   que a geração de código futura já nasça seguindo-as.

## Dica

A `360` é **extensível**: uma nova lente é só mais uma entrada em `sections[]` —
o renderer e o schema não mudam. Se você só precisa de **uma** dimensão, vá
direto na skill específica (`scip-context-budget`, `scip-debloat`,
`scip-maintainability`); a `360` brilha quando o valor está em **ver tudo junto**
num único deck.
